# Contributing to DCIK

Thanks for helping improve DCIK. This guide covers how the repository is laid
out, how to make changes safely, and the one rule that has bitten us before:
**never bump the version by hand.**

## Repository layout

| Path | Purpose |
|------|---------|
| `SKILL.md` | The skill definition Claude Code loads. Version lives in its frontmatter. |
| `perspectives/` | The 178 analytical perspective files (`P0001`–`P0178`). |
| `.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json` | Plugin install manifests — what `/plugin install` reads. Each carries a version. |
| `MANIFEST.json` | SHA-256 hashes of every distributed file (integrity check). Generated, never hand-edited. |
| `package.json` | Package metadata + canonical version. |
| `desktop/` | A full mirror of the skill for the desktop distribution. Generated from root — never edit directly. |
| `scripts/sync-all.sh` | The single tool that bumps the version and regenerates every derived artifact. |
| `.github/` | CI workflows and scripts (validation, behavioral red-team, signature check). |

## Making a change

1. **Branch** off `main`. Direct pushes to `main` are rejected — a repository
   ruleset requires every change to go through a pull request.
2. **Sign your commits.** CI verifies commit signatures. Configure signing
   (`git config commit.gpgsign true` with an SSH or GPG key) before committing.
3. **Make your edit** (see the specific workflows below).
4. **Run `bash scripts/sync-all.sh`** to regenerate derived artifacts, then
   commit everything it changed.
5. **Open a PR.** CI runs content validation, manifest verification, the
   behavioral red-team (on changed perspectives), and the signature check.

### Adding or editing a perspective

- Perspective files live in `perspectives/` and are named `P####-slug.md`.
- Edit or add the file under `perspectives/` only. **Do not** touch
  `desktop/perspectives/` — it is regenerated.
- Run `bash .github/scripts/validate-perspectives.sh` to check format locally.
- Run `bash scripts/sync-all.sh` to mirror to `desktop/`, rebuild `MANIFEST.json`
  and `SKILL.zip`, and confirm root/desktop counts match.
- New perspectives are behaviorally scanned by the DeepSeek red-team in CI; keep
  content analytical (questions and methodology), never instructions addressed
  to a model.

### Editing docs (`README.md`, `ARCHITECTURE.md`, `PHILOSOPHY.md`)

- Edit the root copy. `desktop/` copies are regenerated by `sync-all.sh`.
- Run `bash scripts/sync-all.sh` and commit the mirrored changes too.

## Bumping the version — the one rule

DCIK declares its version in **many** files (`package.json`, `SKILL.md`, both
`.claude-plugin/` manifests, the READMEs, `MANIFEST.json`, and the entire
`desktop/` mirror). Editing a subset silently ships an inconsistent version in
the plugin manifests that users install from — and structural validation still
passes, so the mistake is invisible.

**Never hand-edit version strings.** Bump with a single command that moves every
location atomically and then hard-fails if any version disagrees:

```bash
bash scripts/sync-all.sh X.Y.Z      # pass the new version, e.g. one minor bump up
```

The script:
1. Rewrites the version in every source-of-truth file.
2. Mirrors to `desktop/`, updates the README badges, regenerates `MANIFEST.json`
   (+ `desktop/manifest.json`) and `SKILL.zip`, and syncs any locally installed
   skill.
3. **Enforces consistency** — it scans all tracked `*.json`/`*.md` files and
   exits non-zero if any semver other than the target remains. If the script
   exits `0`, the tree is consistent by construction.

Version semantics (from `ARCHITECTURE.md`): new perspectives or orchestrator
changes are a minor bump; file-format/distribution changes are a major bump;
fixes are a patch bump.

## Before you open the PR

```bash
bash .github/scripts/validate-perspectives.sh   # content format
bash scripts/sync-all.sh                        # re-sync + consistency gate (exit 0 = clean)
git status                                       # commit everything sync-all touched
```

If `sync-all.sh` exits non-zero, read its final section — it lists the exact
files and version strings that disagree. Fix them by re-running with the
intended version rather than editing by hand.
