# Assessment B — SQLite vs Postgres for a multi-tenant B2B SaaS (~500 tenants)

## Recommendation
Postgres (managed, shared-schema with `tenant_id` or schema-per-tenant) is the right default for this profile, but the decision is workload-conditional; the honest drivers are the per-tenant write shape and whether cross-tenant queries are core — not the engine itself.

## Key considerations
- The "SQLite vs Postgres" framing smuggles in "one shared database." SQLite's single-writer lock is per file, so database-per-tenant (Turso/libSQL, D1, LiteFS) lets ~500 independent tenants write in parallel. The first decision is the tenancy model; the engine follows.
- Write concurrency: shared SQLite serialises writers (WAL improves throughput/read-during-write, ~10–50k writes/s on NVMe, not writer parallelism); per-tenant SQLite dissolves it for tenant-local writes but hot tenants still bottleneck. Load-bearing unknowns: writes/sec per tenant, burstiness, transaction duration, tenant-size distribution. Postgres MVCC removes the concern.
- Operational execution: Postgres needs connection pooling (PgBouncer) at scale, and shared-DB coupling is real (hot tenants, bloat, vacuum, global migrations affect all). SQLite-per-tenant trades that for ~500-database migration orchestration from day one, which can erase small-team simplicity. Per-tenant file restore is easier than shared-Postgres whole-cluster PITR — material for B2B compliance.
- Economics: at 500 tenants the cost curves differ — managed Postgres is one instance scaled vertically/with replicas (you pay for peak); per-tenant managed SQLite bills closer to usage (cheaper for many idle small tenants, but 500 databases multiply per-DB overhead/monitoring). Neither dominates; model it against tenant-size distribution.
- Second-order/path dependency: the engine shapes the product's option space. SQLite-per-tenant biases toward edge/local-first/offline futures and strong physical isolation (a B2B compliance selling point); Postgres-first biases toward centralised relational assumptions, easy cross-tenant analytics, and a larger hiring pool.
- Temporal: if a few tenants grow into heavy-write/large accounts, shared Postgres absorbs it with familiar tooling; per-tenant SQLite handles many-small growth better but strains on a few-large tail.
- Cross-tenant analytics/admin: shared-schema Postgres makes global reporting/search/admin straightforward; per-tenant SQLite needs fan-out/ETL — often decisive. RLS helps shared-schema isolation at the cost of policy complexity; per-tenant isolates physically.
- Bias check: reflexive "use Postgres" is partly availability/status-quo bias (evidence about defaults, not this workload); the "SQLite renaissance" narrative is social-proof-loaded (blog enthusiasm is not your write profile). Condition the choice on evidence, not fashion.

## Confidence & what would change the answer
Confidence: HIGH that Postgres is the right default given a small team; MEDIUM that it's optimal, because key workload facts are unspecified. Decision rule: default to managed Postgres shared-schema if you want cross-tenant analytics, one operational surface, and standard hiring (most 500-tenant B2B SaaS land here); switch to SQLite/libSQL per-tenant only if tenant writes are light and independent, physical isolation/edge latency is first-order, cross-tenant queries are not near-term, and you will build the many-DB migration/monitoring story from day one. The single fact that most changes the answer: are cross-tenant queries (global reporting, search, admin) core? If yes → Postgres decisively; if no and writes are tenant-local and light → per-tenant SQLite is genuinely competitive. Residual uncertainty: write-rate distribution, cross-tenant-query need, and 3-year tenant-size tail are unspecified.
